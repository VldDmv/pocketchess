## 1. Что это за проект

Шахматы с тремя режимами (Classic, Chess960/Fischer Random, «Floor is Lava»),
в двух обличьях:

- **Desktop** — оффлайн-приложение на Swing (игра против ИИ / на одном экране).
- **Online** — веб-сервер на Spring Boot: игра против бота и против живого
  соперника (PvP) через браузер, лобби, рейтинги, профили, импорт/экспорт PGN.

Главный архитектурный принцип: **вся шахматная логика живёт в модуле `core`** и
переиспользуется обоими клиентами. Ни desktop, ни online не дублируют правила
игры — они только «обёртки» вокруг общего движка.

## 2. Архитектура: три модуля Maven

Родительский `pom.xml` (packaging `pom`) с тремя модулями:

```
pocketchess-parent
├── core      (pocketchess-core)    — движок, headless, без UI и сети
├── desktop   (pocketchess-desktop) — Swing UI, зависит от core
└── online    (pocketchess-online)  — Spring Boot, зависит от core
```

- Версии библиотек管ятся импортом BOM `spring-boot-dependencies` в
  `dependencyManagement` родителя.
- `core` теперь имеет собственные тесты (JUnit Jupiter + AssertJ как `test`-зависимости).
- `online` тянет `spring-boot-starter-test` (JUnit/AssertJ/Mockito).

**Почему модульность важна:** баг, найденный/исправленный в `core`, чинит сразу и
desktop, и online. Но и наоборот — неаккуратная правка `core` ломает оба клиента,
поэтому ядро прикрыто тестами (см. §14).

Пакеты `core`:
- `general` — `Game` (фасад), `Board`, `RuleEngine`, `LavaAwareRuleEngine`.
- `pieces` — `Piece` и наследники + `Spot` (клетка).
- `game.moveanalyze` — `Move`, `PlayerMoveService`, `TemporaryMoveHandler`,
  `CastlingUtils`, `AIMoveService`, `TurnFinisher`, проверки (`AttackChecker`,
  `MaterialChecker`), интерфейс `ChessRules`.
- `game.status` — `GameMoveExecutor`, `GameStateManager`, `GameTimeManager`,
  `GameStatusManager`.
- `game.gamenotation` — история ходов, трекинг позиций, форматтер нотации, PGN.
- `game.utils` — `FenUtils`, `PgnUtils`.
- `game.model` — `GameMode`, `GameStatus`, `TimeControl`.
- `gamemode` — `GameModeType`, `Chess960Setup`, `LavaManager`, `LavaEffectService`.
- `ai.*` — поиск, оценка, дебютная книга, сложность.

## 3. Ядро: модель доски и фигур

### Координаты (важно для понимания всего кода)

Доска — `Spot[8][8]` внутри `Board`, индексируется как `[row][col]`:

- `row 0` = 8-я горизонталь (задний ряд чёрных), `row 7` = 1-я горизонталь (белых).
- `col 0` = вертикаль `a`, `col 7` = вертикаль `h`.
- Перевод в шахматную клетку: `file = (char)('a' + col)`, `rank = 8 - row`.
  Пример: `(row=7, col=4)` → `e1`; `(row=0, col=4)` → `e8`.

Эта «перевёрнутость» (row 0 сверху) пронизывает движок — держи её в голове, когда
читаешь генерацию ходов или FEN.

### Классы фигур

`Piece` (абстрактный) → `Pawn`, `Knight`, `Bishop`, `Rook`, `Queen`, `King`.
Каждая фигура знает свой цвет (`isWhite`) и умеет `canMove(board, start, end)` —
геометрия хода без учёта шаха (это базовая «псевдо-легальность» для одной фигуры).
`King` и `Rook` дополнительно хранят `hasMoved` (нужно для прав рокировки).

`Spot` — клетка: координаты + ссылка на `Piece` (или `null`). У `Spot` есть
конструктор копирования, который **глубоко копирует фигуру** (новый объект с тем
же `hasMoved`) — это критично для копий доски в ИИ (см. §15).

`Board` хранит `boxes`, текущую цель взятия на проходе (`enPassantTargetSquare`) и
снимок начальной позиции (`initialBoxes`, через `saveAsInitial()` — нужен для
Chess960 и навигации по истории).

## 4. Ядро: жизненный цикл хода

`Game` — фасад, который оркеструет специализированные сервисы (паттерн «разбили
большой God-класс на роли»):

- `GameStateManager` — чей ход, статус (`ACTIVE`/`CHECK`/`CHECKMATE`/…), режим
  (`PVP`/`PVE`), цвет игрока.
- `PlayerMoveService` — приём и исполнение хода игрока.
- `GameMoveExecutor` — физическое перемещение фигур на доске (+ взятия, рокировка,
  взятие на проходе, превращение), ведёт списки съеденных фигур.
- `TurnFinisher` — пост-обработка: лава-эффекты, обновление статуса, переключение
  хода, тайминг.
- `GameStatusManager` — определяет мат/пат/ничью.
- `GameTimeManager` — часы.
- `GameHistoryManager` + `GamePositionTracker` — история ходов и подсчёт повторений
  (для троекратного повторения).
- `AIMoveService` — ход бота (см. §9).

**Поток одного хода игрока** (`Game.playerMove(...)` → `PlayerMoveService.executeMove`):
1. Проверки: игра активна, режим «live» (не просмотр истории).
2. Определить, не рокировка ли это (две формы — см. §6).
3. Проверить легальность (`RuleEngine.isMoveLegal` / `isCastlingMoveLegal`).
4. `GameMoveExecutor.createAndExecuteMove` — выполнить на доске, вернуть `Move`.
5. Добавить в историю, обновить halfmove-счётчик.
6. Если пешка дошла до края — статус `AWAITING_PROMOTION` (ждём выбор фигуры).
7. Иначе `TurnFinisher.finishTurn` — лава, статус, переключение хода, часы.

## 5. Генерация ходов и make/undo

### `FastMoveGenerator`

Генерирует ходы для стороны, чей сейчас ход. Две публичные точки входа:

- `generateMoves(game)` — «для ИИ»: в режиме лавы дополнительно избегает
  WARNING-клеток (поле `avoidWarnings = true`), потому что поиск ИИ не симулирует
  волны лавы и не должен ходить на клетку, которая вот-вот загорится.
- `generateLegalMoves(game)` — «по точным правилам»: WARNING-клетки проходимы
  (реальный движок блокирует только активную лаву). Используется отображением
  легальных ходов в UI и реиграние PGN.

Алгоритм: сначала псевдо-легальные ходы каждой фигуры, затем фильтр «не оставляет
ли свой король под шахом» через **make/undo**:

```java
game.makeTemporaryMove(move);
if (!game.isKingInCheck(<сторона, что ходила>)) legal.add(move);
game.undoTemporaryMove(move);
```

### `TemporaryMoveHandler` (make/undo)

Делает/откатывает ход «начерно» для поиска ИИ и фильтрации легальности.
Должен быть **идеально обратимым и реентерабельным** (поиск вкладывает make/undo на
много уровней). Рокировку откатывает **атомарно** (см. §15, кейс с пропажей ладьи):
сохраняет короля и ладью, чистит обе целевые клетки, ставит обе на исходные — это
корректно даже когда клетки пересекаются (король f1 ↔ ладья g1).

> Перфт-тесты (§14) — главный страж корректности этого узла.

## 6. Правила: легальность, шах, рокировка

`RuleEngine` (+ `LavaAwareRuleEngine` для копий ИИ) реализует `ChessRules`:
- `isMoveLegal(board, start, end)` — для обычного хода: `piece.canMove(...)` +
  `isKingSafeAfterMove(...)` (временно двигает фигуру и проверяет шах).
- `isCastlingMoveLegal(board, start, kingDest)` — король не двигался, не под шахом,
  клетки прохода не под боем, и **строгая** проверка рокировки.
- `AttackChecker` — атакована ли клетка; `MaterialChecker` — недостаточный материал.

### Рокировка — две формы представления

Тонкое место (особенно в Chess960). Ход короля кодируется одним из двух способов:

1. **king→kingDest** — король на свою конечную клетку: `col 6` (g-file) для
   короткой, `col 2` (c-file) для длинной. Это «стандартная» форма; распознаётся,
   когда король идёт **больше чем на одну клетку** (`|Δcol| > 1`).
2. **king-takes-rook** — король «берёт» свою ладью (king→rook). Единственная форма,
   корректная для **любой** линии короля (в Chess960 король может стоять на b/d/f).

Почему важно: если король на b-линии рокируется в ферзевую, он идёт на c1 — это
**одна** клетка, и форма (1) не распознаётся (`|Δcol| == 1`). Поэтому везде, где
рокировку нужно применить программно (ИИ, реиграние PGN, онлайн-адаптер), её
выражают как king-takes-rook. `Move.chess960RookFromCol` хранит колонку ладьи.

`King.canMove`: для обычной валидации хода — если это «попытка рокировки», но
рокировка нелегальна, **откатывается на проверку обычного одноклеточного шага**
(король на b-линии может просто шагнуть на c1). А вот `isCastlingMoveLegal`
использует **строгий** `King.isCastlingMove` (без отката) — иначе заблокированная
рокировка ошибочно сошла бы за легальную (см. §15).

`CastlingUtils` — общие хелперы (`findCastlingRookCol`, `rookDestCol`,
`castlingDirection`), одно определение для `GameMoveExecutor` и `TemporaryMoveHandler`.

## 7. Варианты: Chess960 и Floor is Lava

`GameModeType` = `CLASSIC | CHESS960 | LAVA`.

### Chess960 (`Chess960Setup`)

Генерирует один из 960 валидных задних рядов по SP-номеру (FIDE-нумерация):
2 слона на разных по цвету полях, ферзь, 2 коня, и в оставшихся 3 клетках —
**ладья-король-ладья** (король всегда между ладьями). `Game.resetGame` для
CHESS960 вызывает `applyChess960Setup`, который ставит белый ряд на `row 7`,
зеркальный чёрный — на `row 0`, и `saveAsInitial()`.

### Floor is Lava (`LavaManager`)

Каждые `LAVA_INTERVAL = 6` полуходов: текущие WARNING-клетки (синие) становятся
LAVA (красные, фигуры на них сгорают, кроме королей), и генерируется новая пара
WARNING. Лава блокирует ход «на» и «через» клетку. Ключевое для онлайна и PGN:
последовательность волн **детерминирована от seed'а** RNG — `LavaManager` хранит
`seed`, умеет `reseed(long)` и `getSeed()`, что позволяет точно воспроизвести
партию при импорте PGN. Снимки состояния лавы ведутся для отмены ходов и навигации
по истории.

## 8. Нотации: FEN, UCI, SAN, PGN

- **FEN** (`FenUtils.generateFEN`) — строка позиции: расстановка по рядам, чей ход,
  права рокировки (X-FEN: буквы файлов ладей, верх=белые), цель взятия на проходе.
  Генерируется чтением доски напрямую.
- **UCI** (`UciMove`, онлайн) — длинная нотация `e2e4`, `e7e8q`. Перевод между
  координатами движка `(row,col)` и шахматными клетками. Рокировка Chess960
  отдаётся клиенту как king→rook (`b1a1`).
- **SAN** — короткая нотация (`Nf3`, `exd5`, `O-O-O`). Генерация/парсинг в
  `PgnUtils` и `ChessNotationFormatter`. Рокировка: короткая ⇔ конечная колонка
  короля `== 6`, иначе длинная (надёжнее, чем сравнивать со стартом — ломалось для
  короля на b-линии).
- **PGN** (`PgnBuilder` пишет, `PgnUtils.loadPgn` читает) — семь тегов + ходы.
  Для не-классики пишутся заголовки `[Variant]`, `[FEN]` (Chess960 back-rank),
  `[LavaSeed]`. Импорт: вытащить заголовки → переинициализировать движок (вариант,
  seed, расстановку) → проиграть ходы через генератор+исполнитель.

## 9. Искусственный интеллект

Пакеты `ai.search`, `ai.evaluation`, `ai.opening`, `ai.difficulty`.

- `AIPlayer.findBestMove(game)`:
  1. `Game gameCopy = new Game(game)` — **глубокая копия** (своя доска, свой
     rule-engine в режиме AI-симуляции). Поиск идёт на копии, оригинал не трогается.
  2. Дебютная книга (`OpeningBookManager`) — если есть книжный ход.
  3. `IterativeDeepeningSearch` поверх `NegamaxEngine` (альфа-бета) с
     `TranspositionTable`, `MoveOrderer`, `QuiescenceSearch`.
  4. Оценка позиции — `PositionEvaluator` (+ `AdvancedEvaluator`,
     `StrategicEvaluator`, `PieceSquareTables`).
  5. По `AIDifficulty` с вероятностью делается «нарочитая ошибка»
     (`MistakeGenerator`) — так регулируется сила EASY/MEDIUM/HARD.
  6. `convertMoveToOriginalGame` переносит выбранный ход с копии на оригинал
     (важно: переносит и `chess960RookFromCol`, иначе рокировка ИИ ломалась).

- `AIMoveService` (desktop) и `ChessEngineAdapter.requestAiMove` (online)
  применяют ход бота. Оба: рокировку — как king-takes-rook; и есть **фолбэк** —
  если исполнитель почему-то отверг выбранный ход, играется первый принятый
  легальный ход, чтобы бот никогда не «завис».

> Урок: копия для поиска должна быть **полностью** глубокой. Незаметная мелочь
> (ссылка на чужой `Spot` в цели взятия на проходе) сделала ИИ «слепым» к en
> passant — см. §15.

## 10. Онлайн-слой: Spring Boot

`OnlineApplication` — точка входа. Слои:

**Web (`web/`)** — контроллеры:
- `HomeController`, `LobbyController`, `GamePageController`, `ProfileController` —
  Thymeleaf-страницы.
- `AuthController` — регистрация/логин.
- `PlayController` (`/api/play/*`) — создать игру vs бот / открытый вызов, join,
  cancel, импорт PGN, выгрузка PGN.
- `GameWebSocketController` — STOMP-эндпоинты ходов/действий.
- `GlobalModelAdvice` — общие данные в модель (рейтинг в шапке).

**Игровой слой (`game/`)**:
- `GameRegistry` — потокобезопасный реестр сессий (`ConcurrentHashMap`).
- `GameSession` — состояние одной партии: игроки, движок (через адаптер), часы,
  история (UCI/SAN/FEN + по-плаевая лава/варнинги), оффер draw/undo/rematch,
  берсерк, presence-таймстемпы.
- `GameService` — оркестратор: создание, ход, бот, часы, форфейт, broadcast.
- `GameView` — иммутабельный снимок состояния, который шлётся клиенту после
  каждого события (всё для отрисовки доски, часов, истории, лавы, кнопок).
- `Player`, `Messages`, `PresenceTracker`.

**Движок-адаптер (`engine/`)**:
- `ChessEngineAdapter` — тонкая headless-обёртка над `core.Game` (всегда `PVP`,
  ход бота дёргается явно, без Swing-таймеров). Даёт `legalMoves()` (в UCI),
  `applyMove(uci)`, `requestAiMove()`, FEN, захваты, лава-клетки, PGN,
  `loadFromPgn`.
- `MoveResult` — результат хода (принят? uci, fen, статус, лава/варнинги).
- `UciMove` — парсинг/сборка UCI.

**Сервисы/домен/репозитории**:
- `EloService` — формула Эло (K=32). `RatingCategory` — в какой «ведро» рейтинга
  идёт партия. `GameHistoryService` — сохраняет завершённые рейтинговые PvP-партии
  и двигает рейтинги. `UserService` — регистрация.
- `domain.User` (per-category рейтинги в `user_ratings`), `domain.GameRecord`.
- `repo.UserRepository`, `repo.GameRecordRepository` (Spring Data JPA).

**Безопасность (`security/`)** — форм-логин + опциональный Google OAuth2
(`SecurityConfig`, `GoogleOAuth2*`, `AppUserDetailsService`, `CurrentUser`).

**Хранилище** — H2 (файловая: `jdbc:h2:file:./data/pocketchess`), Hibernate
`ddl-auto: update` (см. §17 про Flyway).

## 11. Реальное время: WebSocket/STOMP и фронтенд

- `WebSocketConfig` поднимает STOMP поверх SockJS. Клиент подписывается на
  `/topic/game/{id}` (broadcast состояния) и `/user/queue/...` (личные сообщения:
  ошибки, редиректы).
- Ход клиента → STOMP `/app/game/{id}/move` → `GameWebSocketController` →
  `GameService.applyMove`. После применения сервер шлёт обновлённый `GameView` в
  топик; все клиенты партии перерисовываются от **авторитетного FEN** (сервер —
  источник истины).
- Фронтенд: `chessboard.js` + jQuery + SockJS + `@stomp/stompjs` (CDN). Файлы
  `static/js/game.js` (доска, ходы, история-реплей, оверлеи лавы/подсветка,
  промоушен-пикер) и `static/js/lobby.js` (создание игр, список открытых, фильтр).
- Рокировка в UI Chess960: легальные ходы приходят как king→rook, поэтому
  подсвечивается клетка ладьи; для удобства фронт принимает и бросок короля на его
  «рокировочную» клетку (c/g), переводя это в king→rook UCI.
- По-плаевая лава: `GameView` несёт `lavaHistory`/`warningHistory` (списки клеток на
  каждый полуход), чтобы при перемотке истории показывать лаву **той** позиции.

## 12. Доменные концепции онлайна

- **Время и «вёдра» (`LobbyEntry.categorise`)**: оценочная длительность
  `base + 40*increment`; UltraBullet (<30с), Bullet (<3мин), Blitz (<8мин),
  Rapid (<25мин), Classical (≥25мин), Unlimited (только для игр с ботом).
- **Рейтинг по категориям (`RatingCategory`, `User.ratings`)**: стандартные шахматы
  делятся по времени, а Chess960 и Lava — отдельные «пулы». Проигрыш в лаве не
  трогает блиц-рейтинг. В шапке показывается максимум (headline).
- **Берсерк**: до начала своего первого хода, для игр строго меньше 10 минут,
  игрок вдвое урезает свои часы (инкремент остаётся). При чистой победе берсеркер
  получает **+50%** прироста рейтинга от победы (в буллете **+25%**), округление
  вниз. Оба игрока могут берсернуть независимо.
- **Presence/форфейт (`PresenceTracker`, `GameService`)**: считаются STOMP-сессии
  на пользователя; есть debounce (50–500 мс «дребезг» при перезагрузке вкладки) и
  2-минутное окно — если игрок не вернулся, партия засчитывается как проигрыш.
- **Abort**: в первые 30с (пока не сделано 2 полухода) партию можно прервать без
  последствий для рейтинга.

## 13. Модель многопоточности

Сервер ведёт много партий одновременно; ходы прилетают из STOMP-потоков, ходы бота
и часы — из пулов `ScheduledExecutorService`/worker-ов.

**Правило:** все мутации одной партии сериализуются на мониторе её `GameSession`.
- `GameService.withGame(gameId, action)` — найти сессию в потокобезопасном реестре
  и выполнить действие под `synchronized (session)`.
- Колбэки бота/часов/форфейта тоже синхронизируются на сессии.
- Разные партии идут независимо (нет глобального лока), поэтому медленный поиск ИИ
  блокирует только свою партию.
- **Нет вложенных локов сессий** (ни один метод не держит два монитора сразу) —
  значит нет дедлоков. Вспомогательные вызовы (broadcast, lobby-push, персист,
  создание реванша) не берут лок чужой сессии.

> Раньше человеческие действия шли под глобальным локом `GameService`, а
> бот/часы — под локом сессии: смешанные локи над одной доской (гонка) +
> сериализация всех партий. Приведено к единому per-game принципу.

## 14. Тестирование: perft и инварианты

Тесты ядра (`core/src/test/.../engine/`) — главный актив надёжности:

- **`PerftTest`** — *perft* (performance test): счёт числа листьев дерева легальных
  ходов на глубину N. Эталонный способ проверки генератора+make/undo.
  - стартовая позиция: 20 / 400 / 8902 / 197281 (глубины 1–4);
  - «Kiwipete» (классическая стресс-позиция с рокировками/взятием на
    проходе/связками): 48 / 2039 / 97862 (1–3);
  - **make+undo = тождество** для каждого легального хода на случайных партиях
    Classic и Chess960 (гарантия, что поиск/генерация не портят доску).
- **`LegalityConsistencyTest`** — генератор и исполнитель согласованы: каждый
  ход, который генератор счёл легальным, исполнитель принимает (на копии,
  применяя ровно как сервисы — рокировку king-takes-rook).

Онлайн-тесты (`online/src/test`) — `GameServiceLifecycleTest`,
`GameHistoryServiceTest`, `MoveGenerationTest` (chess960-рокировки, отсутствие
зависаний/порчи доски, PGN), `LobbyServiceTest`, `EloServiceTest`, и т.д.

Запуск: `mvn test` (весь реактор) или `mvn -pl core test` / `-Dtest=PerftTest`.

> **Почему perft незаменим:** обычные unit-тесты применяют один ход и смотрят
> результат. Perft же гоняет **миллионы** make/undo и ловит ровно те классы багов
> (рокировка, взятие на проходе, связки, обратимость), которые иначе всплывают
> только в реальной игре в редких позициях.

## 15. Истории багов (учебные кейсы)

Самая полезная для обучения часть — реальные баги и их корни.

1. **Рокировка Chess960 «съедала» ладью.** Корень — в `TemporaryMoveHandler`:
   отмена рокировки сначала ставила короля на его клетку, затирая ладью, которая
   приземлилась туда же (король f1 / ладья g1). Лечится **атомарной** отменой
   (сохранить обе фигуры → очистить целевые → восстановить исходные). Срабатывало
   на **каждой** генерации легальных ходов (а она идёт на каждый broadcast и в
   поиске ИИ), как только соседняя рокировка была доступна — поэтому фигуры
   «пропадали сами по себе».

2. **ИИ был слеп к взятию на проходе.** `Board(Board other)` копировал
   `enPassantTargetSquare` как ссылку на `Spot` **исходной** доски. На копии
   `end == enPassantTargetSquare` никогда не истинно → e.p. исчезал. А поиск ИИ
   идёт на копиях. Лечится ремапом цели на `Spot` **новой** доски. Поймал
   `LegalityConsistencyTest`.

3. **Заблокированная рокировка генерировалась как легальная.** После того как
   `King.canMove` стал откатываться на «обычный шаг» (чтобы король на b-линии мог
   шагнуть на c1), `isCastlingMoveLegal` тоже стал принимать рокировку, у которой
   занята клетка приземления ладьи. Лечится: проверка рокировки в `RuleEngine`
   зовёт **строгий** `King.isCastlingMove`, а не `canMove` с откатом. Поймал
   perft (make/undo терял слона).

4. **Бот «зависал» в Chess960.** Несколько причин: (а) рокировку ИИ применяли как
   king→g/c-file, что не распознаётся для короля на соседней линии; (б)
   `convertMoveToOriginalGame` терял `chess960RookFromCol`; (в) генератор и
   исполнитель изредка расходились. Лечение: применять рокировку как
   king-takes-rook, переносить метаданные ладьи, и **фолбэк** на любой принятый
   ход, чтобы бот всегда отвечал.

5. **Stale-сборка.** Несколько раз баг «не воспроизводился у меня, но был у
   пользователя», потому что в IDE не пересобирался модуль `core`, и запускалась
   смесь нового `online` со старым `core`. Урок: при общем модуле — **clean
   rebuild** (`mvn clean install` / Rebuild Project).

Сквозной вывод: **общий движок надо покрывать perft/инвариант-тестами**, а копии
для поиска делать честно глубокими; иначе тонкие баги (рокировка, e.p.) живут
месяцами и всплывают только в редких позициях.

## 16. Сборка, запуск, конфигурация

```bash
# Собрать всё и поставить core в локальный репозиторий
mvn clean install            # с тестами
mvn clean install -DskipTests

# Запустить онлайн-сервер (http://localhost:8080)
mvn -pl online spring-boot:run     # требует, чтобы core был install'ен

# Только тесты ядра
mvn -pl core test
```

- Java 17, Spring Boot 3.3.5.
- H2 файловая БД в `./data/pocketchess` (+ H2-консоль на `/h2-console`).
- При запуске онлайна из IDE убедись, что `core` пересобран (см. кейс 5 выше).
- Google OAuth2 включается только если заданы соответствующие env-переменные;
  иначе работает обычный форм-логин.

## 17. Что осталось в тех.долге

Закрыто в ходе работы: perft-сетка ядра; согласованность генератор↔исполнитель;
единая per-game модель локов; чистка комментариев.

Осознанно **не сделано** (кандидаты на будущее):

- **Flyway вместо `ddl-auto: update`.** Сейчас Hibernate сам «угадывает» миграции
  (`update` только добавляет, не переносит данные — из-за этого однажды обнулились
  рейтинги при смене схемы `elo` → `user_ratings`). План: добавить `flyway-core`,
  переключить Hibernate на `validate`, описать схему версионированными SQL
  (`V1__init.sql`, далее `V2__…`), для существующей H2 — `baseline-on-migrate` или
  пересоздать файл. Это переход от «как-нибудь подгонит» к «схема — это код с
  историей».
- Косметика: отдельный `error.html`; аккуратная обработка попытки регистрации с
  уже занятым именем (сейчас всплывает 500).
- Симметричное логирование ходов бота (сейчас в INFO-лог пишутся только ходы
  человека — ходы бота видны лишь по `fenBefore` следующего хода).
